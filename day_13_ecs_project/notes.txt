Practical VPC Deployment

Terraform Modules: Senior engineers utilize the official terraform-aws-modules/vpc/aws module instead of writing raw routing components. It reduces human error, condenses hundreds of lines of code, and ensures compliance with AWS best practices.

CIDR & Subnet Mapping: The VPC claims 10.0.0.0/16. Public subnets are explicitly defined (e.g., 10.0.1.0/24), and Private subnets are kept logically separate (e.g., 10.0.101.0/24).

NAT Gateway Configuration: Setting enable_nat_gateway = true automatically provisions the NAT, places it in the public tier, and configures the private route tables to use it for outbound internet access. Setting single_nat_gateway = true is used for cost-saving in non-critical environments (otherwise, AWS builds one per Availability Zone).

Packaging & Storing the Payload
The Code (processor.py): The core ETL logic. In an Enterprise environment, this script is designed to run to completion and exit cleanly.

The Container (Dockerfile): The blueprint that bundles the OS, dependencies, and code into a single immutable artifact, guaranteeing it runs identically locally and in the cloud.

The Registry (aws_ecr_repository): The Terraform resource that provisions the secure cloud storage shelf for the Docker image, complete with automatic vulnerability scanning.

The Deployment Cycle: You authenticate via aws ecr get-login-password, docker build the image locally, docker tag it with the repository URI, and docker push it to the cloud.



Security Group Logic
Attachment: A Security Group must be explicitly tied to a specific vpc_id to prevent it from defaulting to the public AWS network.

Implicit Deny: Omitting the ingress block in Terraform results in a strict "Deny All" inbound posture, rendering the server completely unreachable from outside connections.

Explicit Egress: Terraform automatically strips default outbound permissions. You must explicitly define an egress block with protocol = "-1" and CIDR 0.0.0.0/0 to grant the container outbound internet access to reach external APIs or the NAT Gateway.

Stateful Architecture: Security Groups are stateful. If an outbound request is permitted by the egress rules, the corresponding inbound response traffic is automatically allowed back in, regardless of the strict ingress restrictions.



The ECS Task Definition
The Master Blueprint: The aws_ecs_task_definition resource does not execute code. It is a strict configuration template that tells ECS exactly how to provision the Fargate server.

Fargate Sizing: You must explicitly define cpu and memory at the task level when using Fargate. These exact values determine your precise AWS compute billing.

awsvpc Network Mode: Mandatory for Fargate. It ensures the task is injected directly into your VPC and assigned its own private Elastic Network Interface (ENI).

Container Definitions: A JSON-encoded array that details the exact Docker images to pull (referenced dynamically via the ECR repository URL), whether the container is essential to the task's survival, and the logConfiguration required to route standard output (like Python print() statements) to AWS CloudWatch.


Deploying ECS Fargate
The Cluster (aws_ecs_cluster): The logical grouping for your tasks.

The Task Execution Role: An IAM role required by ECS. It strictly grants the Fargate agent the permissions it needs to pull your private Docker image from ECR and push your Python print() statements to CloudWatch logs.

The Security Group (aws_security_group): Bound directly to the VPC. For backend Data Engineering jobs, it should have an egress (outbound) rule allowing 0.0.0.0/0 so the script can reach the NAT Gateway, and zero ingress (inbound) rules to maintain total isolation.

The Task Definition (aws_ecs_task_definition): The core JSON blueprint. It defines the Fargate launch type, hardware requirements (CPU/Memory), and the exact ECR container image URI. It also configures the AWS VPC network mode (awsvpc), which forces the container to use the Elastic Network Interface (ENI) and Subnet routing rules.


Phase 1: The Secure Foundation (The VPC Vault)
First, you build the logically isolated network so your financial data is physically invisible to the public internet.

1. Create main.tf and define the network:

Terraform
provider "aws" {
  region = "us-east-1"
}

module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.5.0" 

  name = "financial-data-vpc"
  cidr = "10.0.0.0/16"

  azs             = ["us-east-1a", "us-east-1b"]
  public_subnets  = ["10.0.1.0/24", "10.0.2.0/24"]
  private_subnets = ["10.0.101.0/24", "10.0.102.0/24"]

  enable_nat_gateway      = true
  single_nat_gateway      = true 
  map_public_ip_on_launch = false
}
2. Deploy the Network:
Open your terminal and run:

Bash
terraform init   # Downloads the VPC module
terraform apply  # Builds the network (takes ~3 mins for the NAT Gateway)
Phase 2: The Payload (Python & Docker)
Next, you write your ETL logic and freeze it into an unbreakable Linux container.

1. Create processor.py:

Python
import urllib.request
import json

def fetch_exchange_rates():
    print("Fetching live data...")
    url = "https://api.exchangerate-api.com/v4/latest/USD"
    response = urllib.request.urlopen(url)
    data = json.loads(response.read().decode('utf-8'))
    
    usd_to_ngn = data['rates'].get('NGN', 'N/A')
    print(f"SUCCESS: 1 USD = {usd_to_ngn} NGN")

if __name__ == "__main__":
    fetch_exchange_rates()
2. Create Dockerfile (No file extension):

Dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY processor.py .
CMD ["python", "processor.py"]
Phase 3: The Cloud Vault (ECR & Pushing)
You build the secure shelf in AWS and push your frozen container from your Mac up to the cloud.

1. Add the ECR Repository to main.tf:

Terraform
resource "aws_ecr_repository" "financial_etl_repo" {
  name                 = "fintech-exchange-etl"
  image_tag_mutability = "MUTABLE"
  image_scanning_configuration { scan_on_push = true }
}
Run terraform apply to create the empty vault.

2. The Terminal Push Sequence:
Run these four commands sequentially in your terminal (Replace <AWS_ACCOUNT_ID> with your 12-digit number):

Bash
# 1. Log your terminal into AWS Docker
aws ecr get-login-password --region us-east-1 | docker login --username AWS --password-stdin <AWS_ACCOUNT_ID>.dkr.ecr.us-east-1.amazonaws.com

# 2. Build the image on your machine
docker build -t fintech-exchange-etl .

# 3. Slap the AWS shipping label on it
docker tag fintech-exchange-etl:latest <AWS_ACCOUNT_ID>.dkr.ecr.us-east-1.amazonaws.com/fintech-exchange-etl:latest

# 4. Upload the heavy container to the cloud
docker push <AWS_ACCOUNT_ID>.dkr.ecr.us-east-1.amazonaws.com/fintech-exchange-etl:latest
Phase 4: The Execution Engine (ECS Fargate)
Finally, you define the Orchestrator, the Security Group, the IAM permissions, and the Task Definition Blueprint.

1. Add the ECS Infrastructure to main.tf:

Terraform
resource "aws_ecs_cluster" "financial_cluster" {
  name = "fintech-etl-cluster"
}

resource "aws_cloudwatch_log_group" "ecs_logs" {
  name              = "/ecs/fintech-exchange-etl"
  retention_in_days = 7
}

resource "aws_iam_role" "ecs_task_execution_role" {
  name = "fintech_ecs_execution_role"
  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{ Action = "sts:AssumeRole", Effect = "Allow", Principal = { Service = "ecs-tasks.amazonaws.com" } }]
  })
}

resource "aws_iam_role_policy_attachment" "ecs_execution_policy" {
  role       = aws_iam_role.ecs_task_execution_role.name
  policy_arn = "arn:aws:iam::aws:policy/service-role/AmazonECSTaskExecutionRolePolicy"
}

resource "aws_security_group" "fargate_sg" {
  name        = "fintech-fargate-sg"
  vpc_id      = module.vpc.vpc_id
  # Outbound only! Zero inbound rules.
  egress {
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

resource "aws_ecs_task_definition" "financial_etl_task" {
  family                   = "fintech-exchange-etl-job"
  network_mode             = "awsvpc"
  requires_compatibilities = ["FARGATE"]
  cpu                      = "256"
  memory                   = "512"
  execution_role_arn       = aws_iam_role.ecs_task_execution_role.arn

  container_definitions = jsonencode([{
    name      = "financial-etl-container"
    image     = "${aws_ecr_repository.financial_etl_repo.repository_url}:latest"
    essential = true
    logConfiguration = {
      logDriver = "awslogs"
      options = {
        "awslogs-group"         = aws_cloudwatch_log_group.ecs_logs.name
        "awslogs-region"        = "us-east-1"
        "awslogs-stream-prefix" = "ecs"
      }
    }
  }])
}
Run terraform apply one last time to wire the engine together.

Phase 5: The Trigger (Running the Job)
Your entire architecture is standing by. ECS is waiting for your command to pull the image and run it.

To manually pull the trigger from your terminal, you use the AWS CLI run-task command. (Note: You will need to grab the actual Subnet ID and Security Group ID from your AWS Console or Terraform state to run this).

Bash
aws ecs run-task \
  --cluster fintech-etl-cluster \
  --task-definition fintech-exchange-etl-job \
  --launch-type FARGATE \
  --network-configuration "awsvpcConfiguration={subnets=[<YOUR_PRIVATE_SUBNET_ID>],securityGroups=[<YOUR_SECURITY_GROUP_ID>],assignPublicIp=DISABLED}"
The moment you hit enter, AWS boots up a serverless Fargate node, locks it in the Private Subnet, pulls the Docker image from ECR, runs your Python script, routes the traffic through the NAT Gateway, prints the exchange rate to CloudWatch, and destroys the server.

You have just mastered the heaviest lifting in Data Engineering.
