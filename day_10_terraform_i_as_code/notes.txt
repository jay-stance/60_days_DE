there is a difference between how we write codde in terraform (declarative) and how we write in python and javascript (imperative)

In python where if we say create a bucket X, and make it public. if we run this script twice, it would probably create the bucket twice, or override the first one.
in terraform on the other end, you just declare the end state, "there should be a bucket X, and it should be public". it checks aws if the bucket exists, if not it creates it. and if it is, it doesn't create it as the end state has already been reached

Terraform: An Infrastructure as Code (IaC) tool that lets you write readable code (HCL) to automatically deploy cloud resources instead of clicking through the AWS console.

The Workflow: Write Code $\rightarrow$ terraform init $\rightarrow$ terraform plan $\rightarrow$ terraform apply.

The State File (tfstate): A massive JSON file that acts as Terraform's memory. It maps the code you wrote to the actual physical resources living in AWS.

Remote State: The enterprise best practice of storing the tfstate file in a central S3 bucket (instead of a local laptop) so an entire team of engineers can safely collaborate without their code colliding.

HCP Terraform (Terraform Cloud): A paid, managed service by HashiCorp that hosts your Terraform runs and securely stores your State Files for you, replacing the need to build your own S3/DynamoDB backends.

Remote vs Local Execution: * Remote: HashiCorp's servers run your code and hold the state.

Local: Your laptop runs the code, but reaches out to HashiCorp to read/write the shared state using an API Token.

State Locking: Terraform locks the state file during an apply so two engineers can't mutate the infrastructure at the exact same millisecond.

State Versions: HCP keeps a historical log of every state file change, allowing you to audit exactly when an AWS resource was created, modified, or destroyed.


HCP Terraform Billing & Resource Types

Managed Resources: The core metric HashiCorp uses to bill you. It is the total number of physical infrastructure items tracked inside your State File.

resource blocks: Code used to create, update, or destroy infrastructure. These do count toward your managed resource bill. (Note: using loops like count or for_each multiplies the number tracked in the state).

data blocks: Code used purely to read or look up existing infrastructure. These do not count toward your bill.
